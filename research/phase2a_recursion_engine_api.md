## 1. Goal and scope

This document specifies the API and core class/module for a `RecursionEngine` that will live in the `disteval` package alongside `SelfEngine`. It is a **design-only** deliverable (no code changes to the existing repository).

The engine turns the implicit RMDP analogy already present in `SelfEngine` (`disteval/self_engine.py`, lines 31–49) into an explicit, optional decomposition step. It does **not** replace the existing flat RECOVERABLE curriculum; it augments it with sub-task RMDPs for STUCK and deeply RECOVERABLE tasks.

Design constraints (from the research skill, `SKILL.md` lines 86–91):
- Python 3.10+ type hints.
- Compatible with the existing numpy / pandas / scipy dependency stack.
- No new runtime dependencies unless justified.
- Grounded in the actual repository; file paths and line numbers are cited.

## 2. Core data classes

All new data classes are placed in a new module `disteval/recursion_engine.py`. They use the same dataclass style already established in `disteval/self_engine.py` (lines 69–134) and `disteval/right_tail.py` (lines 113–158).

### 2.1 `PhaseBoundary` — entry/exit signal for one sub-task

A `PhaseBoundary` records the structural step range that forms a sub-task. It mirrors the role of an RMDP entry/exit node pair (Phase 1B mapping, `research/phase1b_disteval_mapping.md`, lines 184–192).

```python
from dataclasses import dataclass, field
from typing import Optional
from .trajectory_monitor import TrajectoryFeatures


@dataclass
class PhaseBoundary:
    """One contiguous phase in a parent trajectory, treated as an RMDP entry/exit pair."""

    entry_step: int
    exit_step: Optional[int] = None
    label: str = ""
    confidence: float = 0.0
    tool_signature: tuple[str, ...] = field(default_factory=tuple)
    entry_features: Optional[TrajectoryFeatures] = None
    exit_features: Optional[TrajectoryFeatures] = None
    is_predicted_exit: bool = False   # True if exit_step was predicted (not observed)

    @property
    def n_steps(self) -> int:
        """Number of tool-call steps in the phase."""
        if self.exit_step is None:
            return 0
        return max(0, self.exit_step - self.entry_step)
```

**Defaults:** `exit_step=None` means the phase has not been closed by a confident boundary (e.g., the parent trajectory failed before the next checkpoint). `confidence` is in `[0, 1]` and is derived from `TrajectoryMonitor` prediction probabilities (`trajectory_monitor.py`, `PatternMatch.p_high`, lines 75–87).

### 2.2 `SubTask` — a callable sub-RMDP component

A `SubTask` is the disteval equivalent of an RMDP component `Mᵢ` (Phase 1B mapping, `research/phase1b_disteval_mapping.md`, lines 40–43). It has a unique ID, a parent task, a structural boundary, and an incremental reward attributed to reaching the boundary.

```python
@dataclass
class SubTask:
    """Description of one sub-task generated by the RecursionEngine."""

    sub_task_id: str
    parent_task: str
    label: str
    phase: PhaseBoundary
    reward_delta: float = 0.0
    weight: float = 1.0
    child_ids: list[str] = field(default_factory=list)
    parent_id: Optional[str] = None
    depth: int = 0

    # Decomposition provenance
    source: str = "structural_divergence"   # "structural_divergence" | "checkpoint" | "memory"
    entry_traj_path: Optional[str] = None
    exit_traj_path: Optional[str] = None
```

**Defaults:** `reward_delta` is the incremental score the parent task gains by successfully completing this sub-task (e.g., 0.10 for the JSON-validity checkpoint in `medium-2`). `weight` is used when propagating sub-task scores back to the parent (`recursion_engine.py` will use it in `compute_recursive_gap`). `source` documents how the boundary was discovered.

### 2.3 `RMDPNode` — recursive task tree

`RMDPNode` is the in-memory tree that connects a parent task to its `SubTask` children and, recursively, to their children. It is the call-stack structure made explicit (Phase 1B mapping, `research/phase1b_disteval_mapping.md`, lines 46–47).

```python
from .right_tail import TaskOutcomeProfile
from .self_engine import TrainingPair


@dataclass
class RMDPNode:
    """One node in the recursive task decomposition tree."""

    task: str
    profile: TaskOutcomeProfile
    depth: int = 0
    parent: Optional[str] = None
    sub_task: Optional[SubTask] = None

    children: list["RMDPNode"] = field(default_factory=list)
    phase_boundaries: list[PhaseBoundary] = field(default_factory=list)
    training_pairs: list[TrainingPair] = field(default_factory=list)

    # Recursive right-tail analysis on this node
    recursive_q_star: Optional[float] = None
    recursive_q_bar: Optional[float] = None
    recursive_gap: Optional[float] = None
    recursive_consistency: Optional[float] = None

    decomposed_reason: str = "recoverable"   # "recoverable" | "stuck" | "deep_gap"
    cycle: int = 0

    @property
    def is_leaf(self) -> bool:
        return not self.children

    @property
    def sub_task_ids(self) -> list[str]:
        return [c.sub_task.sub_task_id for c in self.children if c.sub_task is not None]
```

**Defaults:** `children` is empty for leaf nodes (SOLID sub-tasks or tasks that are too shallow to decompose). `recursive_*` fields are populated by `RecursionEngine.compute_recursive_gap()` using a bottom-up traversal.

### 2.4 `RecursionEngineConfig` — tunable knobs

```python
@dataclass
class RecursionEngineConfig:
    """Tunable parameters for the RecursionEngine."""

    max_depth: int = 3
    divergence_confidence: float = 0.70
    max_phase_boundaries: int = 5
    min_monotone_difficulty: float = 0.0
    reward_propagation: str = "weighted_sum"   # "weighted_sum" | "max" | "right_tail"
    enable_decompose_stuck: bool = True
    enable_decompose_recoverable: bool = True
    min_sub_task_score: float = 0.0
    require_checkpoint_alignment: bool = False
    max_stack_nodes: int = 100
    memory_retrieval_k: int = 3
```

**Defaults and rationale:**
- `max_depth = 3` — hard recursion cap to prevent infinite decomposition of STUCK tasks (Phase 1 Master Report, section 6, risk #2; Phase 1C recommended answer Q5).
- `divergence_confidence = 0.70` — a `PatternMatch.p_high` above 0.70 is treated as confident HIGH; below 0.30 as confident LOW (mirroring the `0.35 / 0.65` uncertainty band in `TrajectoryMonitor.check()`, `trajectory_monitor.py` lines 498–503).
- `max_phase_boundaries = 5` — limits how many sub-tasks are generated per parent task; matches the five-checkpoint structure of `medium-2` (`tasks/medium-2/tests/test.sh`, referenced in `phase1b_disteval_mapping.md` lines 255–263).
- `min_monotone_difficulty = 0.0` — the monotone difficulty check: each sub-task must be strictly easier than its parent. For a task with a normalized difficulty score, a child must have a lower score. Set to `> 0` to require a minimum margin.
- `reward_propagation = "weighted_sum"` — how sub-task scores are combined into the parent recursive gap. Alternatives are `max` (best sub-task dominates) and `right_tail` (reuse `right_tail_analysis` on the flattened sub-task scores).
- `enable_decompose_stuck = True` — whether STUCK tasks (`q_star == 0`) are decomposed at all.
- `enable_decompose_recoverable = True` — whether high-gap RECOVERABLE tasks are further decomposed.
- `min_sub_task_score = 0.0` — minimum score a sub-task must be able to contribute before it is kept.
- `require_checkpoint_alignment = False` — when True, sub-task boundaries must align with explicit test-suite checkpoints (future Phase 2B feature; default False because the current codebase has no checkpoint API).
- `max_stack_nodes = 100` — safety cap on the total number of RMDP nodes generated in one run.
- `memory_retrieval_k = 3` — number of `TrajectoryMemory` retrievals per sub-task (`self_engine.py` uses `k=3` for whole-task retrieval, line 448).

## 3. `RecursionEngine` class API

### 3.1 Constructor

```python
class RecursionEngine:
    def __init__(
        self,
        monitor: TrajectoryMonitor,
        memory: TrajectoryMemory,
        store: RecordStore,
        config: Optional[RecursionEngineConfig] = None,
        agent_name: str = "agent",
        model_name: str = "unknown",
    ) -> None:
        ...
```

**Why these dependencies?**
- `TrajectoryMonitor` — provides the structural divergence signal used to derive entry/exit boundaries (`trajectory_monitor.py`, `check()` lines 488–524; `self_engine._find_divergence_step`, lines 553–575).
- `TrajectoryMemory` — provides retrieval of similar sub-task demonstrations at matching structural depth (`trajectory_memory.py`, `retrieve()` lines 214–288; Phase 1B mapping, lines 195–206).
- `RecordStore` — stores the raw `EpisodeRecord` rows used to compute sub-task right-tail profiles (`records.py`, lines 19–43).
- `RecursionEngineConfig` — optional tuning (defaults above).
- `agent_name` / `model_name` — mirror the fields of `SelfImprovementPlan` (`self_engine.py`, lines 112–114).

### 3.2 Public methods

```python
    def decompose_task(
        self,
        profile: TaskOutcomeProfile,
        traj_paths: list[tuple[float, str]],
        depth: int = 0,
        parent: Optional[str] = None,
    ) -> RMDPNode:
        """
        Decompose a single task into a tree of RMDP sub-tasks.

        Args:
            profile: right-tail outcome profile for the task.
            traj_paths: list of (score, trajectory_path) for this task.
            depth: current recursion depth (0 = root task).
            parent: optional parent task identifier.

        Returns:
            An RMDPNode whose children are the generated sub-tasks.
        """
        ...

    def decompose_stuck_tasks(
        self,
        report: RightTailReport,
    ) -> dict[str, RMDPNode]:
        """
        Decompose all STUCK tasks in a RightTailReport.

        Returns a dict mapping task name -> RMDPNode. Tasks that cannot be
        decomposed (e.g., no trajectories exist) are omitted.
        """
        ...

    def decompose_recoverable_tasks(
        self,
        report: RightTailReport,
        min_gap: float = 0.30,
    ) -> dict[str, RMDPNode]:
        """
        Decompose high-gap RECOVERABLE tasks that exceed a gap threshold.

        Args:
            report: right-tail report produced by SelfEngine.run_cycle().
            min_gap: only decompose tasks with gap >= min_gap.

        Returns:
            dict mapping task name -> RMDPNode.
        """
        ...

    def find_phase_boundaries(
        self,
        steps: list[dict],
        start_step: int = 0,
        end_step: Optional[int] = None,
    ) -> list[PhaseBoundary]:
        """
        Segment a trajectory into candidate sub-task phases using the monitor.

        Uses the same post-hoc divergence logic as SelfEngine._find_divergence_step
        (self_engine.py, lines 553-575) but returns all confident boundaries instead
        of the first one.
        """
        ...

    def slice_parent_trajectory(
        self,
        traj_path: str,
        boundary: PhaseBoundary,
    ) -> list[dict]:
        """
        Extract the tool-call steps belonging to one sub-task phase.

        The returned slice is suitable for memory ingestion, prompt generation,
        and (in Phase 2B) environment generation.
        """
        ...

    def build_sub_task_profile(
        self,
        sub_task: SubTask,
        scores: list[float],
        model: str = "agent",
    ) -> TaskOutcomeProfile:
        """
        Run right_tail_analysis on a sub-task score vector.

        Reuses the existing task_outcome_profile() logic (right_tail.py, lines 163-204).
        """
        ...

    def compute_recursive_gap(
        self,
        node: RMDPNode,
    ) -> float:
        """
        Bottom-up recursive gap computation for an RMDPNode.

        Returns the parent-level gap implied by the children's sub-task gaps.
        The propagation rule is controlled by config.reward_propagation.
        """
        ...

    def to_curriculum_items(
        self,
        nodes: list[RMDPNode],
    ) -> list[TaskImprovement]:
        """
        Convert a list of RMDP nodes into TaskImprovement items for SelfEngine.

        Items are marked with kind="decomposed" and carry their sub_task list
        in a new `sub_tasks` field on TaskImprovement.
        """
        ...

    def run(
        self,
        report: RightTailReport,
        curriculum: list[TaskImprovement],
    ) -> list[TaskImprovement]:
        """
        Main entry point: decompose STUCK and high-gap RECOVERABLE tasks and
        append the resulting sub-task items to the curriculum.

        Returns a new list; the input list is not modified.
        """
        ...
```

### 3.4 How the engine is instantiated from and called by `SelfEngine`

The change is additive: `SelfEngine` keeps its current constructor and `run_cycle()` signature, and gains an optional `recursion_engine` collaborator.

**Proposed change in `SelfEngine.__init__()`** (`disteval/self_engine.py`, lines 252–265):

```python
    def __init__(
        self,
        store: RecordStore,
        job_dirs: list[str],
        agent_name: str,
        model_name: str,
        monitor: Optional[TrajectoryMonitor] = None,
        memory: Optional[TrajectoryMemory] = None,
        recursion_engine: Optional[RecursionEngine] = None,   # NEW
    ):
        ...
        self.recursion_engine = recursion_engine
```

**Proposed change in `SelfEngine.run_cycle()`** (`disteval/self_engine.py`, lines 375–435):

```python
    def run_cycle(self, cycle: Optional[int] = None) -> SelfImprovementPlan:
        ...
        # Existing flat curriculum
        curriculum = []
        for profile in report.priority_tasks:
            curriculum.append(self._build_task_improvement(profile))
        curriculum.sort(key=lambda x: x.priority_score, reverse=True)

        # NEW: run recursion engine if attached
        if self.recursion_engine is not None:
            curriculum = self.recursion_engine.run(report, curriculum)

        self._attach_predicted_gains(curriculum, report)
        ...
```

**Convenience constructor extension in `SelfEngine.from_job_dirs()`** (`disteval/self_engine.py`, lines 311–361):

```python
    @classmethod
    def from_job_dirs(
        cls,
        job_dirs: list[str],
        agent_name: str = "agent",
        model_name: str = "unknown",
        tasks_dir: str = "tasks",
        enable_recursion: bool = False,          # NEW
        recursion_config: Optional[RecursionEngineConfig] = None,  # NEW
    ) -> "SelfEngine":
        ...
        instance = cls(...)
        if enable_recursion:
            instance.recursion_engine = RecursionEngine(
                monitor=instance.monitor,
                memory=instance.memory,
                store=instance.store,
                config=recursion_config or RecursionEngineConfig(),
                agent_name=agent_name,
                model_name=model_name,
            )
        return instance
```

**Rationale:** This keeps `SelfEngine` fully backward-compatible. Existing callers that do not pass `enable_recursion=True` get the exact same behavior as today. The recursion engine is an optional plugin that reuses the same `monitor`, `memory`, and `store` instances already built inside `SelfEngine`.

## 4. New `TaskImprovement` fields

To carry sub-task information through the existing curriculum, `TaskImprovement` gets two new optional fields (`disteval/self_engine.py`, lines 81–100):

```python
@dataclass
class TaskImprovement:
    ...
    # NEW: recursive decomposition data
    sub_tasks: list[SubTask] = field(default_factory=list)
    rmdp_node: Optional[RMDPNode] = None
```

When `sub_tasks` is non-empty, the `to_dict()` method of `SelfImprovementPlan` (`disteval/self_engine.py`, lines 169–213) serializes each sub-task as:

```json
{
  "sub_task_id": "medium-2::phase-1",
  "parent_task": "disteval/medium-rest-client",
  "label": "HTTP client runs without error",
  "phase": {
    "entry_step": 0,
    "exit_step": 3,
    "confidence": 0.82,
    "tool_signature": ["write_file", "run_shell_command"]
  },
  "reward_delta": 0.10,
  "weight": 1.0,
  "depth": 1,
  "source": "structural_divergence"
}
```

The `kind` field remains one of `"solid" | "recoverable" | "stuck"` for flat items and becomes `"decomposed"` for items produced by `RecursionEngine`. This is a new value, so the `CURRICULUM_FORMAT.md` spec will need a one-line update (Phase 2B/2C task, not this deliverable).

## 5. Minimal code example

```python
from disteval.self_engine import SelfEngine
from disteval.recursion_engine import RecursionEngine, RecursionEngineConfig

# 1. Load a disteval job directory with recursion enabled.
engine = SelfEngine.from_job_dirs(
    job_dirs=["jobs/run_C/disteval-run-C"],
    agent_name="Codex CLI",
    model_name="openai/o4-mini",
    enable_recursion=True,
    recursion_config=RecursionEngineConfig(
        max_depth=3,
        divergence_confidence=0.70,
        reward_propagation="weighted_sum",
        enable_decompose_stuck=True,
    ),
)

# 2. Run one improvement cycle.
plan = engine.run_cycle(cycle=1)

# 3. Inspect the plan.
print(plan.summary())
plan.save("disteval_output/recursion_plan.json")

# 4. Inspect sub-task decomposition for a specific task.
for item in plan.curriculum:
    if item.sub_tasks:
        print(f"Task {item.task} decomposed into {len(item.sub_tasks)} sub-tasks:")
        for st in item.sub_tasks:
            print(f"  - {st.sub_task_id}: {st.label}  reward_delta={st.reward_delta:.2f}")
```

In this example the recursion engine will:
1. Receive the flat `RightTailReport` from `SelfEngine.run_cycle()` (`self_engine.py`, line 394).
2. Identify STUCK tasks and high-gap RECOVERABLE tasks.
3. For each, use `TrajectoryMonitor.check()` (`trajectory_monitor.py`, line 488) to find confident structural boundaries.
4. Build `RMDPNode` trees and append `TaskImprovement` items with `kind="decomposed"`.
5. Return the augmented curriculum to `SelfEngine`, which then attaches predicted gains and builds the final `SelfImprovementPlan`.

## 6. Backward-compatibility considerations

### 6.1 No breaking changes to existing public APIs

- `SelfEngine.__init__` gains one optional argument (`recursion_engine`); all existing positional/keyword calls remain valid.
- `SelfEngine.from_job_dirs` gains two optional arguments (`enable_recursion`, `recursion_config`); default values preserve current behavior.
- `SelfEngine.run_cycle()` return type is unchanged (`SelfImprovementPlan`).
- `TaskImprovement` gains two optional fields with safe defaults; serialization via `to_dict()` remains backward-compatible when the fields are empty.
- `RightTailReport`, `TaskOutcomeProfile`, `TrajectoryMonitor`, `TrajectoryMemory`, and `RecordStore` are **not** modified by this design.

### 6.2 Required new public API surface

- New module: `disteval/recursion_engine.py` (not written yet; to be implemented in Phase 4).
- New imports: `RecursionEngine`, `RecursionEngineConfig`, `SubTask`, `RMDPNode`, `PhaseBoundary`.

### 6.3 Required updates to existing files (Phase 2C / Phase 4)

- `disteval/self_engine.py`:
  - Accept `recursion_engine` in `__init__`.
  - Add `enable_recursion` / `recursion_config` to `from_job_dirs`.
  - Call `recursion_engine.run(report, curriculum)` inside `run_cycle()`.
  - Extend `TaskImprovement` dataclass and `to_dict()` for `sub_tasks` / `rmdp_node`.
- `CURRICULUM_FORMAT.md`: document `kind="decomposed"` and the `sub_tasks` array.
- `disteval/trajectory_monitor.py` (optional in Phase 2, required in Phase 2C):
  - Add `entry_step` / `exit_step` to `PatternMatch` or expose a new `find_phase_boundaries()` method.
  - The recursion engine can work initially with the existing `check()` + `_find_divergence_step` logic, so this is not a hard dependency for Phase 2A.

### 6.4 Behavior changes when recursion is enabled

- A curriculum may contain extra items of `kind="decomposed"`.
- The `n_recoverable` count reported in `SelfImprovementPlan` remains the count of flat RECOVERABLE tasks from `right_tail_analysis`. The recursion engine adds decomposed items, but does not change the underlying taxonomy counts.
- `predicted_total_gain` includes the decomposed items because `SelfEngine._attach_predicted_gains()` runs on the augmented curriculum.

## 7. Relationship to existing disteval concepts

The following table grounds the new API in existing code (extending the Phase 1B mapping, `research/phase1b_disteval_mapping.md`, lines 36–57):

| RMDP / disteval concept | Existing code | New API element |
|---|---|---|
| Task / component MDP | `right_tail.TaskOutcomeProfile` (`right_tail.py`, L116–132) | `RMDPNode.profile` |
| Entry/exit nodes | `TrajectoryMonitor.check()` prediction divergence (`self_engine.py`, L553–575) | `PhaseBoundary.entry_step` / `exit_step` |
| Recursive call / box | `TrajectoryFeatures.n_exec` (`trajectory_monitor.py`, L53) | `SubTask.source == "structural_divergence"` |
| Call stack | `extract_tool_sequence()` (`trajectory_monitor.py`, L139–147) | `RMDPNode.children` (tree) |
| Exit value `V(ex)` | `q_star` (`right_tail.py`, L122) | `RMDPNode.recursive_q_star` |
| Sub-MDP Q-value | `consistency` κ (`right_tail.py`, L124) | `RMDPNode.recursive_consistency` |
| Memory backup | `TrajectoryMemory.retrieve()` (`trajectory_memory.py`, L214–288) | `RecursionEngine._retrieve_sub_task_memory()` |
| Convergence | `κ → 1.0` (`self_engine.py`, L45–49) | `RecursionEngine.compute_recursive_gap()` |

## 8. Open questions for Phase 2B / 2C

1. **Environment generation schema:** Should a sub-task be represented as a full RL environment (states, actions, rewards, transitions) or as a DPO-ready trajectory slice? Phase 2B should produce a JSON schema that extends or replaces the current curriculum format (`CURRICULUM_FORMAT.md`, lines 54–74).
2. **Checkpoint alignment:** The `require_checkpoint_alignment` flag is currently False. Should the recursion engine eventually read `tests/test.sh` checkpoint metadata to label `PhaseBoundary.label` and `SubTask.reward_delta` explicitly? This requires a new task adapter.
3. **Multi-exit vs 1-exit:** The `medium-2` test suite has five checkpoints (`phase1b_disteval_mapping.md`, lines 255–263). Should the engine always decompose multi-exit tasks into chains of 1-exit sub-RMDPs, or support multi-exit nodes with a convergence warning?
4. **Sub-task scoring:** When no explicit sub-task score is available, should we estimate it from the parent score and the checkpoint weights, or should we run the agent on the sub-task in isolation (Phase 4)?
5. **Reward propagation rules:** The current design proposes three rules (`weighted_sum`, `max`, `right_tail`). Phase 2B should validate which rule best matches the actual training_sim dynamics (`training_sim.py`, `apply_training_effect`, lines 199–295).
6. **Memory retrieval by sub-task:** `TrajectoryMemory` currently indexes whole trajectories. Phase 2C should decide whether to add `SubTrajectoryEntry` to `MemoryEntry` or keep retrieval at the whole-task level and let the recursion engine slice afterward.
7. **Cross-agent sharing:** If agent A is SOLID on a sub-task, can its trajectories be used as reinforce targets for agent B on the same sub-task? This is flagged as not yet addressed in the Phase 1 Master Report (section 6, risk #6).
8. **Cycle persistence:** The `sub_tasks` field enables cross-cycle persistence (Phase 1 Master Report, section 4, Q6). Should the engine load the previous cycle's decomposition and reuse it, or recompute from scratch each cycle?
9. **Termination safety beyond depth:** Is `max_depth` enough, or do we also need a `max_total_nodes` cap and a `min_sub_task_score` floor to stop decomposing trivial phases?
10. **API placement:** Should `RecursionEngine` live in `disteval/recursion_engine.py` or be a nested class inside `disteval/self_engine.py`? This deliverable proposes a separate module for clarity, but Phase 2C should confirm import ergonomics.

## 9. Summary of deliverable

This document defines:
- A new `RecursionEngine` class in `disteval/recursion_engine.py` with constructor, public methods, and type hints.
- Four new data classes: `PhaseBoundary`, `SubTask`, `RMDPNode`, and `RecursionEngineConfig`.
- Configuration defaults and their rationale, including `max_depth=3`, `divergence_confidence=0.70`, and `reward_propagation="weighted_sum"`.
- How `RecursionEngine` is instantiated from `SelfEngine` (optional collaborator) and called inside `SelfEngine.run_cycle()`.
- A minimal user-facing code example.
- Backward-compatibility analysis: no breaking changes to existing public APIs; only additive optional fields and a new module.
- Open questions for Phase 2B (environment generation schema) and Phase 2C (prototype integration plan).

No code files were modified; this is the Phase 2A design deliverable only.
